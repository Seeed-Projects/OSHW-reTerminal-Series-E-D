# Contributing Guide

This guide explains what contributors need to do when adding or changing
firmware examples for the reTerminal E-Series Firmware Hub.

The short version:

- Contributors add or update the example source code and its local docs.
- GitHub Actions detects which example changed.
- Only changed firmware is rebuilt.
- Unchanged firmware keeps its previously published latest version.
- Each changed firmware gets a new date-based version.
- Every firmware release contains a full latest package for all examples.

## How the automation works

```
examples/<Example>/          Contributor changes example source
        |
        v
GitHub Actions               Detects which example folder changed
        |
        v
Changed firmware only         Builds just the changed firmware targets
        |
        v
gh-pages branch              Stores firmware/<id>/<date-version>/
        |                    Updates firmware/versions.json
        v
GitHub Release               Publishes a full latest package:
                             changed firmware from the new build,
                             unchanged firmware from its previous latest version
```

The workflow is intentionally incremental. A change inside
`examples/ePaper-Voice-Memo/` rebuilds only the Voice Memo firmware targets. It
does not rebuild RTC, deep sleep, display demos, or any other unchanged example.

## Version rules

Firmware versions are tracked per firmware ID.

Example:

```
firmware/ePaper_VoiceMemo_E1001/2026.06.07/
firmware/ePaper_VoiceMemo_E1001/2026.06.12/
firmware/RTC_PCF8563/2026.06.07/
```

Rules:

- The first build of a firmware on a date uses `YYYY.MM.DD`.
- If the same firmware is rebuilt again on the same date, the next version is
  `YYYY.MM.DD.1`, then `YYYY.MM.DD.2`, and so on.
- The web page reads `firmware/versions.json` and defaults to the newest version
  for the selected firmware.
- The automation no longer creates a separate `latest` version for new builds.
- Existing legacy `latest` firmware may be copied once into a date version so
  old published firmware remains available without rebuilding it.

## What contributors need to do

### Add a standard Arduino example

Create a folder under `examples/` where the folder name and `.ino` filename match
exactly:

```
examples/
  WiFi_Scanner/
    WiFi_Scanner.ino
    README.md
```

The matching folder and `.ino` name is required by Arduino. GitHub Actions can
auto-discover this standard shape and build it as a default Arduino firmware
target for E1001, E1002, E1003, and E1004.

### Update an existing example

Edit files inside that example folder only:

```
examples/<Example>/
```

When the pull request is merged, GitHub Actions rebuilds the firmware targets
mapped to that folder and gives those firmware targets a new version. Other
examples keep their previous versions.

### Add a PlatformIO or multi-device example

PlatformIO examples and examples that need per-device build flags should keep
their build metadata in the automation registry managed by the project. Do not
edit the workflow file directly.

For those examples, open the pull request with:

- the example source code,
- a short example `README.md`,
- the intended supported devices,
- any required build environments or compile flags.

The project automation will use that metadata to decide which firmware IDs to
build.

## What contributors should not edit

Do not manually edit generated firmware output:

- `firmware/versions.json`
- `firmware/catalog.json`
- `firmware/<id>/*/manifest.json`
- `firmware/<id>/*.bin`
- `boot_app0.bin`
- the `gh-pages` branch
- GitHub Releases

Do not manually add rows to a release changelog. Release notes are generated by
GitHub Actions.

Do not edit `.github/workflows/build-and-deploy.yml` just to add a normal
example. The workflow is shared infrastructure.

## Local compile check

Before opening a pull request, make sure the example compiles locally when
possible.

One-time setup:

```bash
arduino-cli config init
arduino-cli config add board_manager.additional_urls \
  https://raw.githubusercontent.com/espressif/arduino-esp32/gh-pages/package_esp32_index.json
arduino-cli core update-index
arduino-cli core install esp32:esp32
```

Compile a standard Arduino example:

```bash
arduino-cli compile \
  --fqbn "esp32:esp32:XIAO_ESP32S3:FlashSize=8M,PartitionScheme=default_8MB,PSRAM=opi" \
  ./examples/WiFi_Scanner
```

If your example depends on a Seeed display library, sensor library, or
PlatformIO environment, mention that in the pull request description.

## Pull request checklist

- The example folder name and main source filename match.
- The code does not contain API keys, tokens, passwords, or private Wi-Fi
  credentials.
- Required secrets are documented with placeholders, not real values.
- The example has a short README if setup steps are not obvious.
- The intended supported devices are listed in the pull request description.
- Local compile was run, or the reason it could not be run is explained.

## After merge

After the pull request is merged into `main`, GitHub Actions will:

1. detect the changed example folder,
2. build only the changed firmware target or targets,
3. write a new date-based version for those firmware targets,
4. update `firmware/versions.json` for the web page,
5. update `firmware/catalog.json` so auto-discovered standard examples can appear on the web page,
6. deploy the Firmware Hub to `gh-pages`,
7. create a GitHub Release that includes a full latest firmware package for all
   examples.

No contributor-side release step is required.
